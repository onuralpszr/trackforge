# Commit and pull request guidelines

These rules apply to every commit and pull request in this repository, whether the
work is human written or assisted by AI tools.

## Commit messages

Use Commitizen-style Conventional Commits and sign off every commit:

```bash
git commit -sm "type(scope): subject"
```

- The `-s` flag adds a `Signed-off-by` trailer. It is required on every commit.
- Format: `type(scope): subject`.
- Allowed `type` values (from `.cz.toml`):
  `build`, `chore`, `ci`, `docs`, `feat`, `fix`, `perf`, `refactor`, `revert`, `style`, `test`.
- `scope` is the tracker or area you touched, for example:
  - `feat(deep_ocsort): add appearance gated association`
  - `refactor(trackers): share kalman track base`
  - `docs(book): document deep oc-sort parameters`
- Keep the subject in the imperative mood and under about 72 characters.
- Put extra detail in the body, separated from the subject by a blank line.

## What must never appear in commit messages or pull requests

These constraints cover commit subjects, commit bodies, PR titles, and PR descriptions:

- No `Co-Authored-By` trailers and no other co-author attribution.
- No "Generated with", "Created by", "Co-authored by an AI", or any similar
  tool-attribution lines.
- No em dash characters. Use a regular hyphen or rephrase the sentence.
- No decorative non-ASCII characters and no ASCII art in commit messages or PR text.
- No robotic filler or sign-off closers (for example "Hope this helps", "Let me know").

## Pull request descriptions

- Write plainly in prose. Describe what changed and why.
- Do not add a `## Summary` header or other boilerplate section headers.
- Keep pull requests small, intentional, and focused. Separate refactoring,
  formatting, and functional changes into different PRs where practical.
- Tell us if AI tools assisted the change, as described in `CONTRIBUTING.md`.

## Before re-pushing a branch

`autofix.ci` may push formatting commits to an open PR branch. Always pull or rebase
before you push again so you do not clobber those commits or create a divergence.

## Changelog

`CHANGELOG.md` is generated by `git-cliff` from the commit history (see `cliff.toml`).
Do not edit it by hand. The emoji section headers in the generated changelog come from
the `git-cliff` configuration and are intentional; they are separate from these
plain-text commit message rules.
